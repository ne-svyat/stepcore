# StepCore — Roadmap и состояние проекта

Обновлено: 2026-07-06. Источник правды для Claude Project.
Код: github.com/ne-svyat/stepcore (публичный, подключён коннектором — читать
актуальные файлы оттуда, не из памяти). Текущая версия: **V8.9, versionCode 36**.

## Что это
Полностью офлайн шагомер для Android с максимальной точностью. Без рекламы,
подписок, регистрации, аналитики, облака, сетевых permission. Все данные —
только на устройстве. Пользователь делает в первую очередь для себя.

## Рабочий процесс (НЕ МЕНЯТЬ, проверен ~35 релизами)
- Пользователь не программист, работает только с телефона (Xiaomi 13T Pro, MIUI).
- Claude пишет и проверяет код в СВОЁМ клоне репозитория, но **пушить не может** —
  пуш делает пользователь. Claude выдаёт готовые блоки команд для Termux.
- Файлы создаются в Termux блоками `cat > путь << 'EOF' ... EOF` — полная
  перезапись. Точечные правки — через python3-heredoc с проверкой якорей (assert).
- Сборка: git push → GitHub Actions → artifact `stepcore-apk` → установка APK
  поверх (подпись стабильная: app/release.keystore в репо).
- versionCode/versionName поднимаются через sed каждый релиз.
- Каждый релиз тестируется пользователем в реальности ДО следующего шага.
- Изменения ядра детектора — отдельными изолированными релизами от UI-фич.
- Стиль общения: ФАКТЫ / ВЫВОДЫ / ГИПОТЕЗЫ, прямо, без воды, критика по существу.

## Текущее состояние: V8.9 (versionCode 36)

### Точность ядра (реальные тесты) — ЯДРО ЗАМОРОЖЕНО
- Ходьба: считает каждый шаг. Бег: считает каждый шаг (лог 06.07 подтвердил:
  инт 340-370мс, чистота 85-100%, стабильно).
- Экран выключен: считает аппаратный чип (гибрид), при разблокировке журнал
  пишет "За время блокировки: N шагов".
- Тряска ×3 вида: 0 ложных.

### Известные пределы ядра (НЕ чинить без данных — решать в V9/V10)
- **Тапы по экрану стоя — считаются.** Жёсткий блок по чистоте потока (V7.11) был
  СНЯТ в V8.2: данные показали, что у реальной ходьбы чистота в скользящем окне
  прыгает 25-75%, не держится высоко → блок рвал ходьбу (deadlock повторного
  входа, стопы на 20-30с). Вывод: тап от шага не отличается ни формой одиночного
  пика (проверены амплитуда, ширина, гироскоп, крест-фактор — все перекрываются),
  ни мгновенной чистотой. Решать в V9 (чип классифицирует) или V10 (чистота как
  мягкий фактор Confidence, не блок).
- Ходьба с непрерывным таппингом одновременно — не считается (редкий случай).
- Восстановление после серии тапов ~20-30 с.
- Наклоны/дрыганье ногами сидя — фундаментальный предел акселерометра, план V10.

### Архитектура счёта (гибрид — ключевое решение)
- **Экран включён** → собственный детектор (StepDetector.kt), полная точность,
  разделение WALK/RUN, транспорт, калибровка.
- **Экран выключен** → BroadcastReceiver на ACTION_SCREEN_OFF/ON. При выключении
  MIUI деградирует поток акселерометра (рваные пачки) → детектор ложно уходил в
  TRANSPORT. Поэтому при screenOff детектор отключается, считает аппаратный
  TYPE_STEP_COUNTER (чип работает при спящем CPU). При включении: resetTransient()
  сбрасывает залипшее состояние детектора, запись в журнал.
- Wakelock (PARTIAL) + Doze-whitelist запрос + MIUI-настройки (замок в недавних,
  автозапуск, без ограничений) — см. MIUI_SETUP.md.
- Резерв: досчёт с чипа при реальной смерти сервиса + heartbeat-диагностика
  (CPU спал / датчик молчал / сервис был мёртв) + ловушка падений в журнал.
- Ручной Стоп = осознанная пауза (чип за паузу не досчитывается, "мёртв" не пишется).

### Схема БД (Room, version 2)
- DayRecord (дата, walkSteps, runSteps) — сумма за день, вся история.
- EventRecord (журнал смен режима, timeMs, photoUri-задел под фото).
- **HourRecord (dateHour "yyyy-MM-dd HH", walk, run)** — почасовая агрегация для
  timeline. Миграция 1→2 безопасная (ALTER, данные не теряются). Пишется
  накопительно (ensureHour+addHour), батчится с persistDb. ВАЖНО: почасовые
  данные копятся с V8.8 — за прошлые часы их нет, дневные+ строятся из DayRecord.

### Реализованные экраны и дизайн-система
- **Тема (V8.4):** тёмная (Material3.Dark), палитра чёрный/красный/синий/белый.
  Красный = акцент/главное действие, синий = вторичный, серые оттенки для
  второстепенного (кнопки навигации не красные — "светофор" убран, V8.7).
  Трикстер-эстетика: асимметричные карточки (надорванный угол, card_asym),
  боковые акцент-полосы, sans-serif-black для крупных чисел. Кастомные ttf-шрифты
  нельзя (heredoc = текст, не бинарник) — только системные.
- **Главный (V8.5-8.7):** двухцветное кольцо прогресса (ходьба синяя / бег красный,
  вместе = прогресс к цели, при достижении фон зеленеет), число+%+км/ккал внутри
  круга, легенда, карточка "ВЧЕРА" (шаги/км/ккал/разница), крупная Старт/Стоп,
  навигация в ряд, технику (калибровка/лог/диагностика) свёрнута в "Инструменты".
- **Статистика (V8.4, V8.9):** рекорды карточками (лучший день/бег/серия/всего),
  heatmap за год (5 уровней красный→зелёный от дневной цели, тап=день),
  "ГОД В ЦИФРАХ" (шаги/км/~часы/активный месяц/среднее; часы — оценка по шагам).
- **Timeline (V8.8):** двухцветные столбцы (ходьба низ, бег верх, высота ∝ шагам),
  адаптивные масштабы: Сегодня/Вчера по часам, 7/30 дней, Год по неделям, Всё по
  месяцам. 20 интервалов из идеи пользователя сведены к 6 осмысленным (остальное —
  плохой UX и пустые данные).
- **Профиль (V8.6, V8.9):** вес/рост/возраст/пол/**цель шагов** свёрнуты в
  раскрывающийся блок "ДАННЫЕ" (авто-раскрыт если пусто). Цель питает кольцо и
  хитмап. Паспорт карточками (всего/дистанция/дней/лучший день).
- **История:** журнал (новые сверху), чекбокс-мультивыбор + плавающая кнопка
  "Копировать выбранные", копия в хронологическом порядке, фильтры, экспорт
  CSV/JSON, удаление через "УДАЛИТЬ ДАННЫЕ".
- **Общее:** UiKit.kt (переиспользуемые карточки/заголовки), портретная ориентация.
- Дистанция/калории (Stats.kt): длина шага от роста, ккал MET-приближение —
  ГРУБЫЕ ОЦЕНКИ до ручной калибровки дистанции (V9).

## Roadmap

### V9 — Гибрид чип+детектор и калибровка дистанции
- ГЛАВНОЕ: рассмотреть архитектуру "аппаратный чип считает ВСЕГДА, детектор
  классифицирует режим (ходьба/бег/транспорт)". Встроенный шагомер Xiaomi работает
  точно — это снимает проблему тапов/наклонов (чип их не считает) и фоновую войну
  с MIUI. Предварительно: 2-3 дня сравнивать вечером итог StepCore со встроенным
  шагомером, разница в % решает.
- Ручная калибровка дистанции 100/500/1000 м → длина шага.

### V10 — Confidence Score
- Множители: стабильность ритма, согласие датчиков, достоверность гравитации,
  доля отбракованных пиков, ЧИСТОТА ПОТОКА (уже измеряется). "Точность: N%" +
  причины. Сюда — мягкое понижение доверия тапам/дрыганью и мультигипотезность.

### V11 — Живая диагностика
- График вертикального ускорения с решениями детектора (частично есть: детальный
  лог по интервалам + счётчики отбраковок в журнале).

### V12+ — из принятого триажа
- Обучение с подтверждением; калибровочные задания-маршруты; гео-слой; фантомный
  преследователь; фото в журнале; фильтры-запросы по журналу ("дни >20к", "бег
  >часа"); виджеты; лого. Дизайн-хвост: "рваная бумага" настоящим краем (нужен
  drawable-ассет); докрутка баланса оттенков кнопок при необходимости.

## Открытые/незакрытые пункты
- Бег после снятия блока чистоты подтверждён логом, полный формальный регресс-набор
  (ходьба 100 / бег 100 / тапы / тряска×3 / дрыганье) с V7.1 так и не прогонялся
  целиком — прогнать перед любым изменением ядра.
- Транспорт (реальная поездка) не проверен — нет машины.
- Точность аппаратного досчёта при экране выкл ~90-100/100 (чип придерживает
  первые шаги) — норма фоновых шагомеров.

## Принципы разработки
- Каждая версия — рабочий APK, проверенный в реальности.
- Ядро меняется только по СОБРАННЫМ ДАННЫМ, не по гипотезам. Промах → диагностика
  (счётчики/детальный лог), не кручение порогов вслепую. Урок раунда 05-06.07:
  три гипотезы фикса тапов (гиро-пол, крест-фактор, блок чистоты) провалились,
  потому что проверялись без записанных данных; сработало только решение по логу.
- Пороги — константы с комментариями, крутятся по тестам, не по теории.
- Разработка гибридная: UI-фичи параллельно со сбором данных по ядру (не скучно +
  видно прогресс + больше времени на тесты).
- Ничего не уходит с устройства.
