# StepCore — источник правды для Claude Project

Обновлено: 2026-07-19. Версия в репозитории: **0.93.0, versionCode 177** (метка релиза v182).

Код: `github.com/ne-svyat/stepcore` — **читать актуальные файлы оттуда, не из памяти и не из старых md проекта**. Файлы 00–011 в проекте описывают древний замысел и годятся только как история.

## 1. Что это

Полностью офлайн шагомер для Android с максимальной точностью. Без рекламы, подписок, регистрации, аналитики и облака. Все данные остаются на устройстве.

Устройство разработки: Xiaomi 13T Pro, MIUI/HyperOS, Termux. Барометра нет.

**Следующий главный трек — обучение и адаптация под конкретного человека** (раздел 6). Дизайн интерфейса закрыт и в переработке не нуждается.

## 2. Как мы работаем (не менять, проверено ~180 релизами)

### Ритм
- **Один релиз — одна задача.** Крупные вещи режутся на под-этапы и выдаются по одному.
- **Сначала обсуждаем замысел, потом пишется код.** Для UI это был макет, для алгоритмов — план измерений.
- **Мелкими кусками экономнее.** Большой патч сжигает токены и почти всегда ломается на мелочи; маленький проверяется целиком и катится за минуту.
- После каждого релиза пользователь ставит APK и говорит результат. Следующий шаг выбирается по факту, а не по плану.

### Роли
- Пользователь не программист, работает только с телефона.
- Claude пишет и проверяет код в своём клоне репозитория; пуш делает пользователь из Termux.
- Все решения по продукту — за пользователем. Claude предлагает, объясняет компромиссы и честно называет риски.

### Доставка кода
**Только скачиваемым `.sh`-файлом.** Пасты кода в Termux запрещены: длинные пасты обрезаются и вешают heredoc.

```
cd ~/stepcore && cp ~/storage/downloads/vXXX.sh . && bash vXXX.sh
```

Каждый скрипт обязан содержать:
- guard «не корень репо», guard «есть незакоммиченные правки», guard «уже применён» (идемпотентность);
- guard предыдущей версии — маркер того релиза, поверх которого патч рассчитан;
- python3-патчи с assert-якорями и проверкой уникальности якоря;
- бамп versionCode/versionName;
- валидацию (см. ниже), `git add app` (никогда `-A`), commit + push.

Падение любой проверки = репозиторий не тронут.

### Обязательная валидация в каждом скрипте
- **Баланс скобок** в изменённых `.kt` (со стриппером строк и комментариев).
- **Баланс `save()`/`restore()`** в файлах отрисовки — иначе едет вся канва.
- **XML well-formed** через `xml.dom.minidom` при правке разметки.
- **Полный баланс тегов** в разметке (открыл-закрыл по всему файлу).
- **Сверка id**: каждый `R.id.*` из кода существует в разметке; дублей id нет.
- **Порядок ключевых элементов** в разметке, если композиция важна.
- Специальные проверки под конкретный патч (например: «шкалы переехали, а не задвоились»).

### Уроки, оплаченные поломками
- **Отступы в якорях.** `rindex("        <LinearLayout")` попадает ВНУТРЬ отступа вложенного элемента (в 12 пробелах есть подстрока из 8). Якорь всегда с переводом строки: `"\n        <LinearLayout"`. Эта ошибка уронила v169 — приложение не запускалось.
- **Границы блока — обходом вложенности**, а не «до следующего похожего тега». Перемещаемый фрагмент обязан быть тегово-сбалансированным, иначе XML остаётся валидным, но дерево пересобирается неправильно.
- **Long и Float не смешивать.** `System.currentTimeMillis() - CONST_F` не компилируется. Сборку это уронило (v180 → хотфикс v181). В скриптах стоит проверка на этот случай.
- **Экранирование в heredoc.** `"$line $word"` внутри heredoc превращается в мусор; в Kotlin-строках использовать явную склейку.
- **Незакрытая замена** `s.replace(a, b)` без `,1)` — хронический дефект.
- **Assert на `count(имя)==N`** даёт ложные срабатывания; писать содержательные проверки.
- **`%%` работает только в `.format()`**; в интерполяции `"$x%"` — одиночный `%`.
- **`this` в `lifecycleScope.launch{}`** — это scope, нужен `this@Activity`.

### Сборка и проверка
- В песочнице Claude **нет Android SDK**: настоящий компилятор — только GitHub Actions. Поэтому опечатки в типах вылезают на сборке, и защита от них — накопленные проверки в скриптах.
- Сборка: push → GitHub Actions → artifact APK → установка поверх (стабильный keystore в репо).
- При `401` на push — истёк PAT: новый classic-токен с галкой repo → `git remote set-url origin https://ne-svyat:ТОКЕН@github.com/...`.
- Релизы с миграцией БД: пользователь делает экспорт ДО установки.

### Стиль общения
ФАКТЫ / ВЫВОДЫ / ГИПОТЕЗЫ. Прямо, без воды. Объяснять по шагам «что нажать и что увидишь». Не выдумывать результаты тестов. Ошибку называть ошибкой и чинить, а не заминать.

## 3. Что уже сделано

### 3.1 Ядро точности (неизменный фундамент)
- **Чип — единственный счётчик.** `TYPE_STEP_COUNTER` считает всегда; `StepDetector` только классифицирует WALK/RUN/TRANSPORT, ведёт журнал и метки. Guard 1 (shake-вето), Guard 2 (чип снимает залипший TRANSPORT). Регресс V9: 96–98/100 во всех режимах, тапы 0, поездка 0.
- **Различение бег/ходьба по амплитуде.** У пользователя каденс бега и ходьбы совпадает (медианы 523 vs 543 мс), но амплитуда разделяется идеально: ходьба 0.8–7.8 м/с², бег 9.2–21.1. Порог `AMP_RUN_FLOOR = 8f` — середина измеренного зазора.
- **STEP_DETECTOR как источник темпа — приговор окончательный.** MIUI отдаёт события пачками ~775 мс, метка времени = доставка.
- **Карман — ограничение железа.** Акселерометр глохнет, чип врёт по времени. Честно признано, не чинится.
- **Калибровка:** живой прогресс, тактильная обратная связь (экран портит измерение: 18% разброса при взгляде на экран против 3% при взгляде на дорогу), ворота качества — разброс >25% не сохраняется.

### 3.2 Энергия, история, данные
- LCDA + Margaria (груз в массе), Mifflin–St Jeor basal, Total = A + B.
- **История профиля** (`profile_history`): сегментированный расчёт по часам, каждый час считается профилем своего часа. Точка пишется при сохранении Профиля, при каждой калибровке **и при смене груза с главного экрана**.
- **Снапшоты дня заморожены** (`DayRecord`: kcalActive, kcalBasal, distanceM, activeSec). Прошлое неизменно.
- **Бэкап**: экспорт JSON schema 2, импорт только добавлением отсутствующего, идемпотентен, в одной транзакции с `NonCancellable`.

### 3.3 Адаптивный слой (начало обучения — это уже работает)
- **Ручные метки уклона** (`TerrainState`: UP / FLAT / DOWN) кнопками на главном экране. Метка транзиентна, при перезапуске сбрасывается на FLAT.
- **Уклон влияет на калории** по часам: `HourRecord.upSteps/downSteps`, `EnergyModel.INCLINE_UP_FACTOR = 1.5f` (Minetti, ориентировочный, подлежит уточнению по данным), `INCLINE_DOWN_FACTOR = 1.0f` (решение пользователя; спуски всё равно пишутся ради будущей калибровки).
- **Корпус размеченных образцов** `terrain_samples`: `timeMs, label, mode, amp, intervalMs, gyro`. Пишется 1 образец на 20 подтверждённых шагов, только при включённом экране. В бэкап не входит (данные устройства).

### 3.4 Интерфейс (закрыт)
Каменный язык: резные плиты с гвоздями, материалы контуров по вкладкам (камень/молния/канат/огонь/лёд/механика), фактура внутри плит, пыль, лианы, светящиеся разломы. Герой — гранёная гора с пятью алмазами-целями, тропой, молнией, северным сиянием, венцом и багровой луной при ×5. Живые сцены во всех вкладках. Кнопка Старт/Стоп зовёт пульсом, когда счёт не идёт, и молчит, когда идёт.

### 3.5 Реплики («мысли StepCore»)
- `Quips.kt`: **267 строк**, шесть корзин, архитектура «контекст → классификаторы → корзины → движок».
- Первый классификатор — шаги за сегодня: до 7к / 7–12к / 12–20к / 20–30к / 30к+.
- Выдача «мешком»: повторов подряд нет, строка не примелькается, пока не показаны остальные.
- Таймер живёт **только на видимом экране** (`onResume`/`onPause`), интервал 15–60 с. Никаких сервисов и фоновой работы.
- Показ — на пергаментном свитке с полным циклом гибели и возрождения.

### 3.6 Дневная цель
Нижняя граница цели — **7000 шагов** (значит ×5 = 35 000). Зажим стоит при сохранении, при показе поля и при чтении, чтобы подтянулись старые значения.

## 5. Принципы разработки

- **Ядро меняется только по собранным данным.** Второй заход после провала — только с диагностикой. Гипотезы, убитые данными за день, — норма процесса.
- **Данные важнее мнений.** При конфликте «теория ↔ реальные логи» побеждают логи.
- **Экран портит измерение.** Обратная связь для калибровки — тактильная, не визуальная.
- **Проверка качества стоит на сохранении**, а не только на индикации.
- **Пороги — константы с комментарием «откуда число»**.
- **Сообщения об ошибках не выдумывают причину.**
- **Прошлое неизменно**: снимки дней и отрисованные тексты не пересчитываются.
- **Не показывать метрику, которая врёт.** Ничего не уходит с устройства.

## 6. ГЛАВНЫЙ ТРЕК: приложение учится узнавать пользователя

### 6.1 Зачем и что это значит
Сейчас приложение считает точно, но **одинаково для всех**. Цель трека — чтобы StepCore со временем понимал именно этого человека: его походку, темп, длину шага, как он носит телефон, когда он идёт в гору, когда бежит, когда едет в транспорте, и когда данные ему врать не должны.

**Граница, которую не переходим:** обучение никогда не трогает счёт шагов. Чип остаётся единственным авторитетом. Модель влияет на *интерпретацию* (метки, уклон, длина шага, калории, подсказки), а не на сам счёт. Ошибка модели должна ухудшать подсказку, а не статистику.

### 6.2 Что уже есть как фундамент
- Ручные метки уклона — это **разметка от человека**, самый ценный вид данных.
- Корпус `terrain_samples` копится с реальных прогулок.
- История профиля позволяет знать, какие параметры действовали в любой момент прошлого.
- Калибровка даёт эталонные интервалы ходьбы и бега.
- `LocationCalibrator` умеет мерить длину шага по GPS (по явному согласию).

### 6.3 Какие данные собирать (и почему именно эти)

**Очевидные, уже частично есть:**
| Сигнал | Что даёт |
|---|---|
| Вертикальная амплитуда шага (пик, медиана, IQR) | главный разделитель бег/ходьба у этого пользователя |
| Интервал шага (медиана, IQR, дрейф в окне) | темп и его стабильность |
| RMS гироскопа | тряска, положение телефона, отсев ложных шагов |
| Режим детектора (WALK/RUN/TRANSPORT/IDLE) | контекст всего остального |
| Шаги чипа (дельты) | сверка и честность |

**Неочевидные, дают больше всего пользы:**
| Сигнал | Что даёт |
|---|---|
| **Ориентация телефона** (pitch/roll из вектора гравитации) | где телефон: карман, рука, рюкзак. Это объясняет половину «странных» измерений |
| **Ось наибольшей вариации гироскопа** | рука vs карман без ручного выбора |
| **Асимметрия чётных и нечётных шагов** (амплитуда, интервал) | несение груза на одном плече, хромота, подъём/спуск |
| **Автокорреляционный пик сигнала** | насколько ритм регулярен; отличает ходьбу от бытовой тряски лучше порогов |
| **Доля энергии на второй гармонике каденса** | у бега выражена, у ходьбы слабее — второй независимый разделитель |
| **Скорость нарастания амплитуды в начале серии** | старт с места vs продолжение движения |
| **Длительность непрерывной серии шагов** | прогулка или перемещение по квартире |
| **Разброс интервалов на границах серии** | торможение, светофоры, лестницы |
| **Экран включён или нет** | человек, глядящий на цифру, меняет походку — это уже измерено |
| **Время суток и день недели** | утренний маршрут отличается от вечернего; серия дней |
| **Груз и профиль на момент замера** | берём из `profile_history`, а не из текущих настроек |
| **Скорость по GPS (только по согласию)** | золотая метка для длины шага и уклона |

**Чего нет и не будет:** барометра (этажи и уклон по высоте невозможны), микрофона, сети, сторонних сервисов.

### 6.4 Этапы (каждый — отдельный релиз или несколько)

**L1. Расширенный снимок признаков.**
Добавить в корпус недостающие сигналы: ориентацию телефона, асимметрию чётных/нечётных шагов, регулярность ритма, длительность серии, признак «экран включён», версию схемы признаков. Ввести `featureVersion`, чтобы старые образцы не смешивались с новыми при обучении.
*Результат:* корпус, на котором вообще можно чему-то учиться.

**L2. Витрина сессий.**
Группировать образцы в сессии движения (непрерывные серии) и хранить агрегаты: медианы, разбросы, доли режимов, метки пользователя. Обучение работает по сессиям, а не по отдельным шагам — так меньше шума и меньше данных.
*Результат:* компактная таблица, пригодная для анализа прямо на устройстве.

**L3. Первый агент: уклон.**
`InclineAgent` предлагает метку UP/FLAT/DOWN по признакам сессии. Сначала **только предлагает**: показывает подсказку, человек подтверждает или исправляет одним касанием. Каждое подтверждение — новая размеченная точка (активное обучение). Автоматическая метка включается лишь после того, как согласие с человеком стабильно высокое на его собственных данных.
*Результат:* уклон перестаёт быть полностью ручным, а точность калорий растёт на измеренных, а не выдуманных коэффициентах.

**L4. Персональная модель походки.**
По накопленным сессиям уточнять: интервалы ходьбы и бега, амплитудный профиль, коэффициенты длины шага a/b, порог `AMP_RUN_FLOOR` под конкретного человека. Плюс **детект дрейфа**: если медиана дня стабильно уходит от эталона (например, >15% в 7 из 10 дней) — предложить перекалибровку.
*Правила:* учиться только на high-confidence сессиях; предохранители (гиро-капы, транспорт-логика) не обучаются никогда; хранить версии модели и уметь откатываться; прошлое не пересчитывать.

**L5. Агентный слой (перенос идеи MotionLab).**
Агенты — **read-only классификаторы над уже посчитанными числами StepCore**, а не второй счётчик. Каждый отвечает за одну вещь и не лезет в чужую:
- `PatternAgent` — характер дня относительно личной нормы;
- `EnergyAgent` — интерпретация расхода (не пересчёт: авторитет по калориям остаётся у `Stats`);
- `LoadAgent` — влияние груза;
- `TerrainAgent` — рельеф по сессиям;
- `DecisionAgent` — сводит выводы в одну оценку дня.
Регистрация списком, как в наброске MotionLab. Один источник истины: агент не имеет права заново считать то, что уже посчитано ядром — исходный дефект наброска был именно в этом (`EnergyAgent` считал калории без груза).
*Куда встроить:* отдельный модуль `:motionlab` (чистый Kotlin-JVM, без Android) с односторонней зависимостью `:app → :motionlab`. Тогда логику можно гонять тестами в песочнице до установки на телефон. После переноса отдельный репозиторий MotionLab удаляется.

**L6. Реплики становятся умными.**
Новые классификаторы для `Quips`: темп, серия дней, время суток, уклон, необычность дня относительно личной нормы. Архитектура для этого уже готова — добавляется одной строкой в `CLASSIFIERS` плюс своя корзина.

### 6.5 Как проверять, что обучение работает
- **Согласие с человеком:** доля случаев, когда предложенная метка совпала с подтверждённой. Растёт — модель учится.
- **Сверка с чипом** остаётся в журнале как и раньше.
- **Регресс на реальных журналах:** каждое изменение проверяется на записанных днях, а не на синтетике.
- **Правило остановки:** если новая версия улучшает один сценарий и ухудшает другой — она не принимается.

## 7. Хвосты (некритичные, по приоритету)

1. `INCLINE_UP_FACTOR = 1.5f` — ориентировочный коэффициент, заменить измеренным по корпусу.
2. Timeline: активное время в деталке считается на лету, а не из снапшота.
3. Пары-вспышки Бег↔Ходьба на торможении — лечить гистерезисом по времени, только при жалобе.
4. Диагностику STEP_DETECTOR можно удалить (вердикт вынесен).
5. `recentMean` уходит в минус после resetTransient — косметика.
6. Confidence Score: гироскоп оказался вторым разделителем (ходьба 0.6–1.0, бег 1.8–2.7) — готовый вход.
